# specstory-skill — Specification

Single source of truth for what this skill is, how it behaves, and the contracts
it upholds. Keep this updated alongside code changes. Companion docs:
[README](README.md) (user-facing), [ARCHITECTURE](skills/specstory/references/ARCHITECTURE.md)
(file-by-file), [cloud-api](skills/specstory/references/cloud-api.md) (GraphQL).

## 1. Purpose

Make a developer's AI coding session history queryable from inside any agent.
Answer "what was I working on?", produce work digests/standups, search past
sessions, and recap a project's recent AI activity — **local-first** (no key, no
network) with an optional cloud layer.

## 2. Scope & non-goals

- **In scope:** read-only querying/summarizing of SpecStory session data (local
  markdown or cloud); structured recap extraction; cross-tool + plugin packaging.
- **Non-goals:** writing/syncing sessions (that's the SpecStory CLI's job),
  modifying history, an MCP server (possible future addition), authentication UX
  beyond reading an existing key/JWT.

## 3. Data sources (backends)

| Backend | Source | Auth | When used |
|---------|--------|------|-----------|
| **local** (default) | `.specstory/history/*.md` | none | a `.specstory/history` is discoverable |
| **cloud** | SpecStory Cloud GraphQL | `SPECSTORY_API_KEY` or CLI JWT | `--cloud`, or auto when no local history but auth exists |

Selection precedence: `--cloud` > `--local` > auto(local if available, else cloud
if authed, else error). See ARCHITECTURE §"Backend selection".

## 4. Normalized session contract

Both backends MUST return the shape in ARCHITECTURE §"The normalized session
shape". Commands depend only on that shape, never on backend specifics.

## 5. Commands

| Command | Inputs (beyond backend flags) | Output |
|---------|-------------------------------|--------|
| `recap.mjs` | `--per N` (default 1), `--json` | Most-recent session(s) per project + structured recap |
| `digest.mjs` | `--hours N` / `--week`, `--limit`, `--json` | Sessions grouped by repo over a window + recaps + totals |
| `sessions.mjs` | `--project --tool --model --branch --tag --limit` | Sessions grouped by project |
| `search.mjs` | `"<query>"` positional, `--project --limit` | Ranked results w/ excerpts |
| `session-detail.mjs` | `<id\|filename\|name>` `[projectId]`, `--summary`, `--recap` | One session, full content or recap |
| `projects.mjs` | `--limit` | Projects + counts + last activity |
| `introspect.mjs` | (cloud only) | Schema dump to `lib/schema.json` |

**Common backend flags** (all commands): `--local`, `--cloud`, `--root <dir>`,
`--since <iso>`, `--until <iso>`. Every command supports `-h/--help`.

## 6. Behavioral requirements

- **Local-first, zero-config:** with no env and no flags, in a project that has
  `.specstory/history`, every command MUST work without network or key.
- **Keyless cloud convenience:** when `--cloud` is requested with no
  `SPECSTORY_API_KEY`, the skill MUST try the CLI JWT at
  `~/.specstory/cli/auth.json` before failing.
- **Graceful no-source:** when neither backend is available, exit non-zero with a
  message naming both remedies (run inside a SpecStory project / pass `--root`, or
  set up cloud auth + `--cloud`).
- **Idle flagging:** sessions whose `endedAt - startedAt > 12h` MUST be marked
  `⚠️ idle?` in human output and `likelyIdle: true` in JSON.
- **Window filtering:** `--since/--until` applied client-side regardless of backend
  (cloud's server-side date filter is loose).
- **JSON mode:** `--json` (recap, digest) MUST emit a single valid JSON document
  on stdout and nothing else — it is consumed by automation (e.g. coaching crons).
- **No dependencies:** Node.js >= 18 standard library only; no install step.

## 7. SpecStory local file format (parsed contract)

```
<!-- Generated by SpecStory, Markdown vX.Y.Z -->
# <UTC timestamp>
<!-- <Tool> Session <uuid> (<UTC timestamp>) -->
_**User (<ts>)**_  …
_**Agent (<model> <ts>)**_  …
```

Filenames: `YYYY-MM-DD_HH-MM-SSZ[-<slug>].md`. The owning repo directory
(parent of `.specstory/`) is the project name.

## 8. Packaging & distribution

- **Agent Skill** at `skills/specstory/` per [agentskills.io](https://agentskills.io/specification):
  `name` == dir name (`specstory`); `description` ≤ 1024; `license`,
  `compatibility` set; tool hints under namespaced `metadata.*`. Installable via
  `npx skills add kdmonroe/specstory-skill` or manual copy into `~/.claude/skills/`.
- **Claude Code plugin** via `.claude-plugin/plugin.json` +
  `.claude-plugin/marketplace.json`; install with
  `/plugin marketplace add kdmonroe/specstory-skill` then `/plugin install specstory`.
- **Cross-tool:** the same SKILL.md is read by Claude Code, Cursor, Codex, and
  OpenClaw/Hermes (the latter via the namespaced metadata hints).

## 10. Versioning

Skill `version` lives in `SKILL.md` (`metadata.version`), `plugin.json`,
`marketplace.json`, and `package.json` — keep them in lockstep. Current: **3.0.0**
(local-first rewrite; supersedes the cloud-only v1 and the hermes-internal v2).

## 11. Relationship to hermes-deploy

`hermes-deploy/skills/specstory` was a parallel v2.0 fork (cloud-only, with the
`_recap.mjs` parser and `introspect.mjs`). This repo is now the canonical source;
hermes-deploy SHOULD vendor it (submodule / `npx skills add` / sync) rather than
maintain a copy. Hermes' coaching cron calls `digest.mjs --json --cloud` (the
container has no local history); hermes-only metadata stays under `metadata.hermes`.
